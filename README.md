# Determinantal Ideals in Lean 4

[![Lean](https://github.com/WangJiabai/Determinantal-Ideals-Formalization/actions/workflows/lean.yml/badge.svg)](https://github.com/WangJiabai/Determinantal-Ideals-Formalization/actions/workflows/lean.yml)

This repository formalizes in Lean 4 the Gröbner-basis part of
Sturmfels’s theorem for determinantal ideals. For a generic `m × n` matrix
over an arbitrary field, it proves that all `(r + 1) × (r + 1)` generic
minors form a Gröbner basis of the ideal they generate for every monomial
order satisfying `IsAntidiagonalTermOrder`. The order `antiDiagonalLex` is a
concrete instance.

In Lean, `Jr m n r k` is by definition the ideal generated by these minors.
The development does not formalize reducedness or the geometric identification
of this generated ideal with the vanishing ideal of the rank-`≤ r` matrix
variety.

The ambient polynomial ring is the coordinate ring of the generic matrix:

```lean
MvPolynomial (Fin m × Fin n) k
```

The proof combines generic minors, diagonal and anti-diagonal term-order
infrastructure, the Knuth–Robinson–Schensted (KRS) correspondence, the
Doubilet–Rota–Stein straightening law via Swan’s Laplace-product proof, and a
degree-wise Hilbert-function argument. The final theorem uses anti-diagonal
term orders.

## Source Layout

The following files are on the transitive import path starting at
[`Groebner.lean`](MyProject/Determinantalideals/Groebner.lean).

| Core theorem path | Role |
| --- | --- |
| [`DeterminantalIdeal.lean`](MyProject/Determinantalideals/DeterminantalIdeal.lean) | Project-local, Mathlib-style generic-minor and generated-ideal API based on `Matrix.mvPolynomialX`. |
| [`Basic.lean`](MyProject/Determinantalideals/Basic.lean) | Compatibility names and content lemmas for the paper-oriented development. |
| [`Finite.lean`](MyProject/Determinantalideals/Finite.lean) | Finiteness facts for minor indices and fixed-size minor sets. |
| [`Ideal.lean`](MyProject/Determinantalideals/Ideal.lean) | Generated-ideal membership and determinantal-ideal nesting. |
| [`MinorTerms.lean`](MyProject/Determinantalideals/MinorTerms.lean) | Diagonal, anti-diagonal, and permutation terms in determinant expansions. |
| [`DiagonalOrder.lean`](MyProject/Determinantalideals/DiagonalOrder.lean) | Abstract term-order predicates and concrete lexicographic orders. |
| [`Sturmfels_lemma.lean`](MyProject/Determinantalideals/Sturmfels_lemma.lean) | General Mathlib lemmas used in the combinatorial and Hilbert-function proofs. |
| [`Bitableaux.lean`](MyProject/Determinantalideals/Bitableaux.lean) | Young bitableaux, generalized permutations, width, and `Jr`/`GrPlusOne`. |
| [`KRScorrespondence.lean`](MyProject/Determinantalideals/KRScorrespondence.lean) | Forward/reverse KRS, its equivalence, and degree/width compatibility. |
| [`StraighteningLaw.lean`](MyProject/Determinantalideals/StraighteningLaw.lean) | The Doubilet–Rota–Stein straightening development via Swan’s Laplace-product proof, polynomial independence, and quotient bases. |
| [`Groebner.lean`](MyProject/Determinantalideals/Groebner.lean) | Hilbert-function comparison and the final abstract and concrete Gröbner-basis theorems. |

These reusable generic-matrix modules are imported by the top-level aggregate
but are not on the final theorem dependency path:

| Auxiliary generic-matrix API | Role |
| --- | --- |
| [`Eval.lean`](MyProject/Determinantalideals/Eval.lean) | Evaluation of generic minors at concrete matrices. |
| [`ConcreteMatrix.lean`](MyProject/Determinantalideals/ConcreteMatrix.lean) | Concrete-matrix minor compatibility under coefficient maps. |
| [`BaseChange.lean`](MyProject/Determinantalideals/BaseChange.lean) | Base-change compatibility for generic minors and generated ideals. |
| [`Sign.lean`](MyProject/Determinantalideals/Sign.lean) | Auxiliary determinant-sign and minor-matrix bridge lemmas; currently not on the main theorem dependency path. |

The external `Groebner` package supplies the general Gröbner-basis API used
at the final theorem layer.

## Main Results

The recommended descriptive API is:

```lean
Determinantal.KRS.krsEquiv
Determinantal.straightening_law
Determinantal.exists_standardBitableau_basis_determinantalRing
Determinantal.hilbertFunction_detRing_eq_card_monomialExp_width_le
Determinantal.GrPlusOne_isGroebnerBasis_of_isAntidiagonalTermOrder
Determinantal.GrPlusOne_isGroebnerBasis_antiDiagonalLex
```

Declarations beginning with `theorem1_`, `lemma5_`, and `lemma6_` are
correspondence aliases following source-paper numbering; clients should prefer
the descriptive names. `Determinantal.exists_krsEquiv` is an existential
paper-facing wrapper around `Determinantal.KRS.krsEquiv`.

## Build

Install cached dependencies and build the repository with:

```bash
lake exe cache get
lake build
```

The project uses the current package name `MyProject`, so imports are of the
form:

```lean
import MyProject.Determinantalideals.Groebner
```

## Artifact audit

Run the individual artifact checks with:

```bash
bash scripts/check_sources.sh
lake env lean PaperAudit.lean
python3 scripts/check_axioms.py
bash scripts/check_artifact.sh
```

[`Audit.lean`](Audit.lean) uses `#print axioms` to display the transitive axioms
of selected core results. Foundational constants such as `Classical.choice`,
`propext`, or quotient soundness principles do not mean that the project
declares a project-specific axiom. [`check_axioms.py`](scripts/check_axioms.py)
executes this report and enforces the explicit allowlist in
[`audit/axiom_allowlist.txt`](audit/axiom_allowlist.txt).

[`PaperAudit.lean`](PaperAudit.lean) compiles manuscript-facing declaration
names and theorem signatures. [`check_sources.sh`](scripts/check_sources.sh)
scans all tracked Lean sources, including root audit files. The CI workflow
enforces these source-level checks on every pushed revision. Source scanning,
interface compilation, kernel checking, and axiom allowlisting test different
properties.

Generate manuscript metadata or fixed source links with:

```bash
python3 scripts/artifact_metadata.py --format markdown
python3 scripts/paper_links.py --format markdown
```

Publishable permalinks should be generated only from a clean, committed,
frozen revision; the link generator refuses a dirty working tree.
